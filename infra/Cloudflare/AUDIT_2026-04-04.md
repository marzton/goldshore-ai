# Cloudflare Audit — API Worker Bindings + `goldshore.ai` DNS

Date: 2026-04-04
Scope: repo-defined desired state vs canonical Wrangler manifests for Cloudflare workers and DNS.

## Sources audited

- `infra/Cloudflare/desired-state.yaml`
- `infra/Cloudflare/BINDINGS_MAP.md`
- `infra/Cloudflare/config.yaml`
- `infra/Cloudflare/gs-api.wrangler.toml`
- `apps/gs-gateway/wrangler.toml`
- `apps/gs-control/wrangler.toml`
- `apps/gs-mail/wrangler.toml`
- `infra/Cloudflare/gs-web.wrangler.toml`
- `infra/Cloudflare/gs-admin.wrangler.toml`

## Audit summary

Status: **Drift detected** between desired-state documentation and worker/runtime manifests.

### High-priority findings

1. **`gs-api` worker bindings are documented but not declared in Wrangler.**
   - `desired-state.yaml` defines KV (`KV`), D1 (`DB`), R2 (`ASSETS`), and AI (`AI`) bindings for `gs-api`.
   - `infra/Cloudflare/gs-api.wrangler.toml` currently contains commented placeholders only and no active binding blocks.
   - Impact: runtime may fail or rely on UI-only/manual bindings that are not codified.

2. **`gs-api` production route is documented but not declared in Wrangler.**
   - `desired-state.yaml` expects `api.goldshore.ai/*` routing.
   - `gs-api.wrangler.toml` has commented `route` examples only in `env.prod`.
   - Impact: route ownership can drift from repo truth.

3. **Service-binding environment naming is inconsistent across docs/manifests.**
   - Gateway -> API service binding uses `environment = "production"`.
   - API worker environment block is `[env.prod]` (not `[env.production]`).
   - Desired state also claims `environment: production` for service bindings.
   - Impact: high risk of cross-worker binding mis-targeting.

4. **`ops.goldshore.ai` DNS record missing in desired-state DNS table.**
   - `gs-control` route is `ops.goldshore.ai/*`.
   - DNS desired-state records list web/admin/gw/api/mail but no `ops` record.
   - Impact: control worker route may exist without corresponding declarative DNS.

### Medium-priority findings

5. **Preview and secondary hostnames are not consistently represented.**
   - Bindings map includes preview/secondary hosts (`preview`, `admin-preview`, `api-preview`, `gw-preview`, `agent`).
   - Desired-state DNS currently models only core production hosts.
   - Impact: incomplete infra-as-code coverage; harder reconciliation.

6. **Control worker secret binding documentation exists but not codified in Wrangler.**
   - Bindings map expects `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` secrets.
   - `apps/gs-control/wrangler.toml` defines non-secret vars only.
   - Impact: ambiguous source-of-truth for secrets (CLI/UI/manual).

## Recommended remediation order

1. **Codify `gs-api` prod route and required bindings in `infra/Cloudflare/gs-api.wrangler.toml`.**
2. **Normalize service-binding environment naming (`prod` vs `production`) across:**
   - `apps/gs-gateway/wrangler.toml`
   - `infra/Cloudflare/desired-state.yaml`
   - docs (bindings map / runbooks).
3. **Add `ops` DNS record to `desired-state.yaml` (or explicitly document why it is intentionally omitted).**
4. **Decide canonical policy for preview DNS and codify it in desired state (include/exclude explicitly).**
5. **Document secret-binding management approach (Wrangler secrets commands vs dashboard) in runbook and apply consistently.**

## Notes

- This is a **repository configuration audit only**; no live Cloudflare API verification was performed in this pass.
- Build-token policy references are present for `gs-web` and `gs-api` wrangler manifests and in runbook guidance.
