### 2. Sensitive Value Left as Empty Placeholder in Production wrangler.toml

**File:** `wrangler.toml` (lines 14–15, repo root)

The root `wrangler.toml` (which configures `gs-gateway` for the repo-root context) has empty string placeholders for production secrets:

```toml
CLOUDFLARE_ACCESS_AUDIENCE = ""
CLOUDFLARE_TEAM_DOMAIN = ""
```

**Impact:** If this file is ever used accidentally as a deploy target (e.g., `wrangler deploy` run from the repo root without specifying a config), the gateway deploys with **no Access audience validation and no team domain** — effectively disabling all auth enforcement. The canonical config in `apps/gs-gateway/wrangler.toml` has correct values, but this root file is a footgun.

**Fix:** Delete the root `wrangler.toml` or add explicit `# DO NOT DEPLOY — legacy root config` guards.

### 3. `env.prod` Missing AGENT Service Binding in gs-gateway

**File:** `apps/gs-gateway/wrangler.toml` (lines 79–114)

Every worker defines two production environment aliases (`[env.prod]` and `[env.production]`) that are supposed to be equivalent, but the gs-gateway `[env.prod]` block is **missing the `AGENT` service binding** that exists in `[env.production]`:

```toml
# [env.production] — has AGENT binding ✅
[[env.production.services]]
binding = "AGENT"
service = "gs-agent"
environment = "prod"

# [env.prod] — AGENT binding is ABSENT ❌
```

The deploy workflow (`deploy-gs-gateway.yml`) runs `wrangler deploy --env prod`, so the **production deploy is missing the AGENT binding every time CI runs**.

**Fix:** Add the missing `AGENT` service binding to `[env.prod]` in `apps/gs-gateway/wrangler.toml`, or consolidate to a single `[env.production]` and update all deploy workflows to use `--env production`.

## 🟠 HIGH (Fix Before Next Release)

### 4. Non-Standard pnpm Install in gs-api and gs-admin Deploy Workflows

**Files:** `.github/workflows/deploy-gs-api.yml` (line 23-24), `.github/workflows/deploy-gs-admin.yml` (line 92-93)

Both use `npm install -g pnpm@9` while gs-control, gs-gateway, and gs-agent all use `pnpm/action-setup@v4`:

```yaml
# Inconsistent approach in gs-api and gs-admin
- name: Install pnpm
  run: npm install -g pnpm@9
```

**Impact:** Doesn't respect the `packageManager` field in `package.json`, no pnpm binary cache benefit, risk of version drift between deploy workflows.

**Fix:** Standardize all deploy workflows to use `pnpm/action-setup@v4` with `run_install: false`.

---

### 5. Floating `cloudflare/pages-action@v1` Tag in gs-web Deploy

**File:** `.github/workflows/deploy-gs-web.yml` (line 133)

```yaml
uses: cloudflare/pages-action@v1
```

This is a mutable tag. `deploy-gs-admin.yml` correctly pins to a SHA (`@f0a1cd58cd66095dee69bfa18fa5efd1dde93bca`).

**Impact:** Supply chain risk — a compromised or breaking update could silently affect the public gs-web deploy.

**Fix:** Pin `cloudflare/pages-action` in `deploy-gs-web.yml` to the same SHA used in `deploy-gs-admin.yml` or the latest verified SHA.

---

### 6. `ci.yml` pnpm Setup Misaligned; Recursive Build Hits Stale Workspaces

**File:** `.github/workflows/ci.yml`

CI uses `corepack enable` + `corepack prepare` (not `pnpm/action-setup@v4`). The `setup-node@v6` cache setting `cache: "pnpm"` depends on the pnpm action being installed first. Additionally, `pnpm -r build` runs against all workspaces including the stale compat packages (`apps/web`, `apps/admin`).

**Fix:** Switch CI to `pnpm/action-setup@v4`. Consider excluding stale workspaces from recursive builds.

---

### 7. `internal.ts` Uses `Bindings: any` — Type Safety Hole in a Privileged Route

**File:** `apps/gs-api/src/routes/internal.ts` (line 8)

```typescript
const internal = new Hono<{ Bindings: any }>();
```

All other API route files use `Hono<{ Bindings: Env; Variables: Variables }>`. The internal route — which handles inbox status and DNS sync status — uses `any`.

**Impact:** TypeScript cannot catch mistyped binding names (`c.env.CONTROL_LOG` instead of `c.env.CONTROL_LOGS`, etc.). Errors only surface at runtime.

**Fix:** Replace `Bindings: any` with the proper `{ Bindings: Env; Variables: Variables }` generic.

---

### 8. Stale Compatibility Workspaces Still Active in pnpm Workspace

**Files:** `pnpm-workspace.yaml`, `apps/web`, `apps/admin`

The workspace glob `apps/*` includes the documented-stale `apps/web` (`@goldshore/web-compat`) and `apps/admin` (`@goldshore/admin-compat`) compatibility aliases. Per `CURRENT_MONOREPO_STATE.md`, these are "stale alias still present."

**Impact:** Dependency install, recursive builds, and turbo pipelines all run against dead packages. Increases confusion for contributors and automated agents. CI builds may fail or produce noisy output due to stale packages.

**Fix:** Remove `apps/web` and `apps/admin` directories entirely, or add exclusion globs to `pnpm-workspace.yaml`.

### 9. gs-agent Deploy References External Infra wrangler Config via Fragile Path

**File:** `.github/workflows/deploy-gs-agent.yml` (line 54)

```yaml
run: pnpm exec wrangler deploy --env production --config ../../infra/Cloudflare/gs-agent.wrangler.toml
```

All other workers use their own `wrangler.toml`. The infra config sets `main` as `../../apps/gs-agent/src/index.ts` — a relative path that only resolves correctly from `apps/gs-agent/`. This is fragile: if the `working-directory` ever changes, the path silently breaks.

**Fix:** Consolidate gs-agent to use `apps/gs-agent/wrangler.toml` directly.

### 10. `v1` API Stubs Return Hardcoded Fake Data in Production

**File:** `apps/gs-api/src/index.ts` (lines 183–190)

```typescript
v1.get('/agents', (c) => c.json({ agents: ['agent-alpha', 'agent-beta'] }));
v1.get('/models', (c) => c.json({ models: ['gpt-4', 'claude-3'] }));
```

These stub routes return fabricated data. They are also **not covered by `requirePermission()`**.

**Fix:** Remove these stubs or replace with real implementations. Add permission guards if kept.

### 11. `gs-control` Automation Routes Lack Role-Based Authorization

**File:** `apps/gs-control/src/index.ts` (lines 95–98)

```typescript
app.post("/dns/apply", async (c) => c.json(await DNS.sync(c.env)));
app.post("/workers/reconcile", async (c) => c.json(await Workers.reconcile(c.env)));
app.post("/pages/deploy", async (c) => c.json(await Pages.deploy(c.env)));
app.post("/access/audit", async (c) => c.json(await Access.audit(c.env)));
```

These are protected by Cloudflare Access JWT but **not** by `isAuthorizedRole`. The `/system/sync` route in the same file correctly checks roles; these four do not.

**Impact:** Any authenticated Cloudflare Access user — not just admins — can trigger DNS sync, worker reconciliation, pages deployments, and access audits.

**Fix:** Apply `isAuthorizedRole` checks matching the `/system/sync` pattern to all four routes.

### 12. Inconsistent `CLOUDFLARE_TEAM_DOMAIN` Format in gs-gateway Environments

**File:** `apps/gs-gateway/wrangler.toml`

- `[env.production]` / `[env.prod]`: `CLOUDFLARE_TEAM_DOMAIN = "gsl-ops"` (short subdomain)
- `[env.preview]`: `CLOUDFLARE_TEAM_DOMAIN = "goldshore.cloudflareaccess.com"` (full FQDN)
- All other workers use: `CLOUDFLARE_TEAM_DOMAIN = "goldshore.cloudflareaccess.com"`

One of these formats is wrong. Cloudflare Access JWKS endpoints require the full team domain FQDN.

**Fix:** Verify which format works for JWKS resolution and align all gateway environments to use the same format other workers use.
